# 磁盘空间占满问题分析报告

## 给小白的技术科普版

---

## 一、先理解几个基本概念

### 1.1 内存（RAM）是什么？

**内存好比是你的"工作桌"**。

想象你在办公室工作：
- **内存 = 工作桌**：桌子大小有限，但你正在处理的文件都放在桌上，取用速度快
- **磁盘 = 文件柜**：容量很大，但取文件要走过去翻找，速度慢

当你的工作桌（内存）堆满了文件，你就需要把暂时不用的文件挪到文件柜（磁盘）里。

```
┌─────────────────────────────────────────────────────────────┐
│                    你的电脑工作原理                           │
│                                                             │
│   ┌──────────────┐         ┌───────────────────────────┐   │
│   │   内存 RAM    │   快!   │      硬盘 Disk            │   │
│   │  （工作桌）   │ ←─────→ │      （文件柜）           │   │
│   │   容量有限    │         │       容量巨大            │   │
│   │   速度快      │         │       速度慢              │   │
│   └──────────────┘         └───────────────────────────┘   │
│        16GB                      3TB (你的/workspace)       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 Swap（交换空间）是什么？

**Swap 是"临时仓库"**。

当工作桌（内存）真的满了，但你还想继续工作怎么办？
→ 你在文件柜里开辟一块"临时仓库"（Swap），把桌上的文件暂时挪进去。

```
┌─────────────────────────────────────────────────────────────────┐
│                     Swap 的作用                                  │
│                                                                 │
│   内存 16GB                     磁盘上的 Swap 文件               │
│   ┌────────────┐               ┌──────────────────────┐        │
│   │ 正在用的   │               │ 内存溢出的内容       │        │
│   │ 程序数据   │ ──挪出去──→   │ /workspace/swap.img  │        │
│   │            │               │      48GB            │        │
│   └────────────┘               └──────────────────────┘        │
│                                                                 │
│   问题：这个"临时仓库"竟然建在别人的文件柜里（NFS网络存储）！     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**关键理解**：
- Swap 本质上是用磁盘空间来"假装"内存
- 因为磁盘速度比内存慢100倍以上，所以用Swap时电脑会变慢
- Swap应该建在**本地硬盘**（你自己的文件柜），而不是网络硬盘

### 1.3 NFS（网络文件系统）是什么？

**NFS 是"远程文件柜"**。

想象你的公司有个中央仓库：
- 你通过**网络**连接到这个仓库
- 读取文件需要走"网络走廊"，比本地文件柜慢
- 这个仓库是**共享**的，所有人都用它

```
┌──────────────────────────────────────────────────────────────────────┐
│                      你的存储架构                                     │
│                                                                      │
│    你的电脑                           中央仓库（NFS服务器）            │
│    ┌─────────────────────┐           ┌────────────────────────────┐ │
│    │ 本地硬盘             │           │ 192.168.210.250 服务器     │ │
│    │ /dev/sda            │           │                            │ │
│    │ - 46GB 系统         │           │  ├── share_downloads (10T) │ │
│    │ - 18GB home         │           │  ├── share_gpu (10T)       │ │
│    │                     │           │  └── share (3T) ←─你的空间 │ │
│    │                     │  网络连接  │       /workspace           │ │
│    │                     │ ═══════════│                            │ │
│    └─────────────────────┘           └────────────────────────────┘ │
│                                                                      │
│    ❌ 错误：Swap文件建在 NFS 上！                                    │
│    /workspace/swap.img (48GB)                                       │
│                                                                      │
│    这相当于：把你的"临时仓库"建在遥远的中央仓库里                     │
│    → 每次存取都要走网络走廊，超级慢！                                │
│    → 还占用了共享仓库的宝贵空间！                                    │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 为什么会占满 2.3TB？

这是因为：

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  问题的数学原理                                                      │
│                                                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                             │   │
│  │  串口数据流速：假设 1MB/秒（高波特率场景）                    │   │
│  │                                                             │   │
│  │  程序运行时间：23小时45分钟 = 约 85500秒                     │   │
│  │                                                             │   │
│  │  理论累积量：1MB × 85500秒 = 85.5GB                          │   │
│  │                                                             │   │
│  │  但实际更大，原因：                                          │   │
│  │  - 数据可能是二进制格式，占用更多                            │   │
│  │  - Go语言的内存分配有开销                                    │   │
│  │  - 多个客户端连接可能同时累积                                │   │
│  │                                                             │   │
│  │  这就是为什么 kill 后释放了约 2.3TB 空间！                   │   │
│  │                                                             │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 三、问题的双重根源

### 根源一：代码设计缺陷（已修复）

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  问题代码：Mock 类型的无限累积                                       │
│                                                                     │
│  // server.go 中的 MockServer                                       │
│  type MockServer struct {                                           │
│      data []byte  // ← 这个字段无限增长！                           │
│  }                                                                  │
│                                                                     │
│  func (m *MockServer) Broadcast(data []byte) {                      │
│      m.data = append(m.data, data...)  // 每次调用都追加，不清理    │
│  }                                                                  │
│                                                                     │
│  这就像：                                                            │
│  - 你每次收快递都堆在客厅                                            │
│  - 从来不扔掉任何快递                                                │
│  - 客厅堆满了就挪到车库                                              │
│  - 车车库堆满了就挪到仓库                                            │
│  - 仓库也被你塞满了！                                                │
│                                                                     │
│  修复方案：                                                          │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                                                             │   │
│  │  添加容量限制：                                              │   │
│  │                                                             │   │
│  │  type MockServer struct {                                   │   │
│  │      data          []byte                                   │   │
│  │      MaxBufferSize int  // 新增：最大容量限制（默认1MB）     │   │
│  │  }                                                          │   │
│  │                                                             │   │
│  │  func (m *MockServer) appendDataWithLimit(newData []byte) { │   │
│  │      m.data = append(m.data, newData...)                    │   │
│  │                                                             │   │
│  │      // 超过限制就丢弃旧数据                                 │   │
│  │      if len(m.data) > m.MaxBufferSize {                     │   │
│  │          m.data = m.data[len(m.data)-m.MaxBufferSize:]      │   │
│  │      }                                                      │   │
│  │  }                                                          │   │
│  │                                                             │   │
│  │  这就像：客厅最多放100件快递，超过就扔掉最旧的               │   │
│  │                                                             │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

### 5.2 需要手动修复的系统配置

```
┌─────────────────────────────────────────────────────────────────────┐
│                                                                     │
│  Swap 配置修复（需要 root 权限）                                     │
│                                                                     │
│  操作步骤：                                                          │
│                                                                     │
│  1. 关闭当前 Swap                                                    │
│     sudo swapoff /workspace/swap.img                                │
│                                                                     │
│  2. 删除 NFS 上的 Swap 文件                                          │
│     sudo rm /workspace/swap.img                                     │
│                                                                     │
│  3. 在本地硬盘创建 Swap                                              │
│     sudo fallocate -l 32G /swap.img                                 │
│     sudo chmod 600 /swap.img                                        │
│     sudo mkswap /swap.img                                           │
│     sudo swapon /swap.img                                           │
