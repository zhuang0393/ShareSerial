# 📝 远程串口共享工具（ShareSerial）需求设计文档

## 1. 项目概述

### 1.1 背景与目标
在嵌入式与 Android 系统开发中，物理开发板通常连接在某一台特定 PC（或服务器）上。本工程旨在开发一套**跨平台、零配置、无感体验**的串口共享系统。通过网络将一台机器上的物理串口虚拟化到多台远程机器上，使多位工程师可以像使用本地物理串口一样，同时读取 Log 并有序进行 Shell 交互。

### 1.2 核心设计理念
* **无感（Transparent）：** 客户端虚拟出的串口必须在系统底层表现为标准串口（Ubuntu 上的 `/dev/ttyXXX`，Windows 上的 `COMX`），完美兼容既有的全部串口终端和烧录工具。
* **便捷（Zero-Config）：** 支持局域网内服务端自动发现（mDNS/mDNS-SD），客户端无需手动输入 IP 即可一键连接。
* **有序（Ordered）：** 内置输入仲裁机制（抢占/独占模式），防止多人同时输入导致数据乱码。

---

## 2. 系统架构与演进路线

系统采用 **Server / Client** 架构，底层数据传输基于增强型 TCP（支持控制信道与数据信道分离，或采用标准 RFC2217 协议）。

### 🗓️ 阶段开发计划
* **Phase 1（当前重点）：** Ubuntu 服务端（核心引擎） + Ubuntu 客户端（实现 Linux 生态闭环）。
* **Phase 2：** Windows 客户端（对接 Linux 服务端，解决大部分开发人员的跨平台看 Log 需求）。
* **Phase 3：** Windows 服务端（实现全平台对等互通）。

---

## 3. 功能需求详解

### 3.1 Phase 1：Ubuntu 服务端与客户端（核心奠基）

#### 3.1.1 Ubuntu 服务端功能
1. **物理串口动态捕获：**
   * 自动扫描系统中的 `/dev/ttyUSB*`、`/dev/ttyACM*`。
   * 支持动态热插拔识别，板子断电重启后，服务能自动重新绑定。
2. **高性能多路复用广播（One-to-Many）：**
   * 采用事件驱动模型（如 `epoll` 或 Golang / Rust 异步网络库），物理串口数据读入一次，高效广播给所有已连接的客户端。
   * **流量隔离：** 某个客户端网络卡顿不应影响其他客户端接收 Log。
3. **局域网自动广播（服务发现）：**
   * 集成 mDNS 协议。服务端启动后，在局域网内广播自身 IP、端口及当前挂载的串口名称（如 `Pixel_UART_9000`）。
4. **输入仲裁机制（独占控制）：**
   * **读写分离：** 默认所有客户端均有“只读（Read-Only）”权限。
   * **抢占模式：** 客户端可发起 `Acquire Write-Lock` 请求。获得锁的客户端才能向串口写入数据（发送指令），其他客户端处于“等待/被动锁定”状态。支持超时自动释放锁。

#### 3.1.2 Ubuntu 客户端功能
1. **虚拟串口驱动级创建（CUSE/PTY）：**
   * 基于 Linux 伪终端（PTY）技术，在本地 `/dev/` 下创建自定义命名的虚拟串口（如 `/dev/vttyShare0`）。
   * 向系统注册标准 `termios` 属性，确保 `minicom`、`picocom` 等工具能够无缝打开。
2. **一键自动连接：**
   * 扫描局域网内的 ShareSerial 服务端，列表展示可用串口，用户点击“连接”后自动完成网络映射。
3. **状态自愈（断线重连）：**
   * 当网络波动或开发板重启（服务端断开）时，客户端虚拟串口不断开（上层看 Log 软件不崩溃），后台自动尝试重连，恢复后无缝继续输出 Log。

---

### 3.2 Phase 2：Windows 客户端（跨平台扩展）

1. **Windows 虚拟串口驱动集成：**
   * 集成或封装开源的内核级虚拟串口驱动（类比 com0com 机制，或实现一个应用层的命名管道转 COM 桥接器）。
   * 在 Windows 设备管理器中生成真实的 **`COM端口 (COMX)`**。
2. **GUI 桌面管理工具：**
   * 开发一个轻量级的 Windows 任务栏托盘应用。
   * 可视化展现局域网内的 Ubuntu 服务端串口列表，支持“一键挂载为 COM4”/“一键断开”。
3. **完美兼容性：**
   * 必须完美兼容 Windows 平台常用的 `SecureCRT`、`MoboXterm`、`Xshell` 以及各类芯片厂的串口烧录/工具链软件。

---

### 3.3 Phase 3：Windows 服务端（全功能完备）

1. **Windows 物理串口监听：**
   * 利用 Windows API (`CreateFile`, `ReadFile`, `SetCommState`) 捕获本地物理 `COM` 口数据。
2. **对齐 Linux 服务端功能：**
   * 移植 mDNS 服务发现、多路复用网络广播、输入仲裁逻辑至 Windows 平台。

---

## 4. 技术选型建议

为了实现跨平台、高性能、易部署的目标，建议采用以下技术栈：

* **核心开发语言：** **Rust** 或 **Go**
  * *理由：* 均具备极强的跨平台能力。Rust 在处理底层串口（`serialport-rs` 库）和底层网络具有极高的性能和确定性；Go 则开发效率极高，网络库生态成熟。
* **Linux 底层技术：**
  * 服务端直接操作标准 `File` 描述符。
  * 客户端使用 `posix_openpt()` 和 `ptsname()` 创建 PTY 伪终端设备。
* **Windows 底层技术：**
  * 应用层可尝试通过创建虚拟驱动，或在 Phase 2 早期阶段，程序内部打包带签名的 `com0com` 驱动并通过命令行静默调用。
* **服务发现：** 采用 `mDNS`（如 Rust 的 `mdns-sd` 库或 Go 的 `zeroconf`）。
* **UI 框架（若需要）：** **Slint** 或 **Tauri**（轻量、跨平台、内存占用极小，非常适合工具类开发）。

---

## 5. 非功能性需求与体验指标

1. **延迟（Latency）：** 网络传输带来的额外串口延迟应小于 **5ms**，确保 Shell 输入时无明显跟手延迟。
2. **独立性：** 软件打包后应为**单文件绿色版**（Linux 下为一个 Binary 纯命令或单 App，Windows 下为一个 `.exe`），无需用户手动安装复杂的依赖包。
3. **鲁棒性：** 支持高波特率（如 `921600` 或 `1500000`）下的长时间稳定性测试，连续跑 Log 24 小时不断流、不丢包、不内存泄漏。
